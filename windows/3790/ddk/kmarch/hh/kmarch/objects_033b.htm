Object Management

This section describes how objects are managed by the Object Manager, and how drivers and user-mode components interact with these objects. This section includes the following topics:

Object Names

Object Directories

Life Cycle of an Object

Object Handles

The Microsoft® Windows® executive is object-based. Files, devices, synchronization mechanisms, registry keys, and so on, are all represented as objects in kernel mode. Objects are manipulated by the Object Manager, which is a subsystem of the executive. .

Kernel-mode components, including drivers, can reference objects by a pointer to the object . User-mode components can only manipulate objects indirectly, by an opaque data structure called a handle. For more information about handles, see Object Handles.

Objects can also optionally have an object name. User-mode components can use the object's name to open a handle to an object. For more information, see Object Names.

Note that Windows executive objects have no relation to COM objects, or objects in C++ or any other object-oriented programming language.

Examples of kernel-mode objects that are used by drivers include:

  • Device objects (See Device Objects and Device Stacks.)
  • File objects.
  • Symbolic links.
  • Registry keys.
  • Threads and processes.
  • Kernel dispatcher objects, such as event objects and mutex objects. (See Kernel Dispatcher Objects.)
  • Callback objects. (See Callback Objects.)
  • Section objects. (See Section Objects and Views.)
