Registration and Deregistration of NDIS Miniport Drivers with WMI

NDIS automatically registers with WMI each instance of a miniport driver (that is, each functional device object created by NDIS in response to a successful return of MiniportInitialize). After a miniport driver instance is registered as a data provider with WMI, WMI clients can send it queries and sets and register to receive miniport driver status indications. A miniport driver does not have to explicitly register with WMI; NDIS automatically does this for the miniport driver after the miniport driver initializes.

When the miniport driver is unloaded (that is, after the miniport driver's MiniportHalt function returns), NDIS automatically deregisters the miniport driver instance with WMI so that WMI will no longer send queries or sets to the miniport driver.

For each miniport driver instance registered with WMI, NDIS registers GUIDs (globally unique identifiers) that each correspond to a particular OID or a miniport driver status indication. For each miniport driver, NDIS registers GUIDs for a standard set of required OIDs and miniport driver status indications. For more information, see Standard Miniport Driver OIDs Registered with WMI and Standard Miniport Driver Status Registered with WMI. If a miniport driver supports custom OIDs or status indications NDIS also registers GUIDs for these with WMI. For more information, see Customized OIDs and Status Indications.

For connection-oriented miniport drivers, NDIS also registers any named VCs. Only VCs that a call manager, integrated miniport call manager (MCM), or connection-oriented client has named with NdisCoAssignInstanceName are visible to WMI clients. For more information, see Support for Named VCs.
